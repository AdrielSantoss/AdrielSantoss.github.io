O que é uma API

Primeiro, devemos entender o que é uma Api: API é a sigla para Application Programming Interface (em português, Interface de Programação de Aplicações).

Em resumo, API é um meio de comunicação entre dois sistemas, entenda API como um conjunto de regras, protocolos e ferramentas que permite que diferentes softwares se comuniquem e troquem informação.

Quando criamos uma API, basicamente estamos criando um tipo de software que define uma interface para outros sistemas se comunicarem com ele — expondo dados e funcionalidades de forma controlada e padronizada.

Quando falamos "interface" já vem logo na cabeça as interfaces gráficas, esses erros são comuns para quem está começando a entrar no mundo da web, a área da programação possui uma sopa de letrinhas absurda, então, para não ocorrer confusões vamos destacar o que uma API não é:

  • 1. Uma api não é um banco de dados: Uma API acessa um banco de dados, mas ela não é o banco de dados. Api, normalmente não armazenam dados, ela age como um intermediário entre o cliente e a fonte de dados.

  • 2. Uma API não é uma interface gráfica (UI): API significa Interface de Programação, não Interface de Usuário. Ela não tem botões, telas nem layouts — quem interage com ela são outros programas, não pessoas diretamente.

  • 3. Uma API não é um servidor web: Esse é um erro muito comum, vejo muitos programadores falando "äpi é o servidor da aplicação", isso está completamenta errado. O servidor (Express, Django, Laravel e etc) é quem hospeda a API e processa as requisições. A API é o conjunto de regras e endpoints implementados dentro desse servidor.

  • 4. Uma API não é automaticamente REST: Esse é um erro clássico, REST é um entre vários estilos de arquiteturas de API, existem outros como: SOAP, GraphQL, gRPC e etc... Cada um definem suas regras para a interface de comunicação. REST é o estilo mais popular e mais adotado no mercado, por isso, muita gente confunde pensando que api e rest são conceitos equivalentes.

  • 5. API não é backend: A API faz parte do teu backend, mas ela não representa o teu backend. A API é uma camada responsável expor dados e funcionalidades do teu backend. A API sozinha não serve pra nada, ela precisa de um backend e ou uma fonte de dados para expor.

  • 6. Nem toda API é web: Nem toda api é web, ou seja, nem toda api utiliza o HTTP para comunicação, existem Apis de sistemas locais, APIs que permitem a comunicação entre aplicações e o sistema operacional, sem usar rede.

Removendo essas confusões de conceitos no meio do caminho, podemos visualizar o destino com mais clareza, uma API não é nada mais que uma interface de comunicação entre softwares.

O que é uma API REST/RESTFUL

Uma API REST é um tipo específico de API baseada no protocolo HTTP e fundamentada na arquitetura cliente-servidor. REST — sigla para Representational State Transfer — é um estilo de arquitetura criado por Roy Fielding em 2000, na sua tese de doutorado.

Embora o termo “arquitetura REST” seja comum no mercado, tecnicamente REST não é uma arquitetura em si , mas sim um estilo arquitetural composto por restrições aplicadas sobre uma arquitetura cliente-servidor. Ele orienta o design de sistemas distribuídos, mas não define por completo a arquitetura de uma aplicação. Entenda que arquitetura é o projeto e um estilo arquitetural são as regras definidas para construir esse projeto — Rest é exatamente isso, um conjunto de regras que orientam a criação de uma aplicação cliente-servidor.

O estilo arquitetural REST é definido por um conjunto de princípios (ou restrições) que orientam a forma como sistemas distribuídos devem se comunicar. Esses princípios são: cliente-servidor, que promove a separação de responsabilidades; stateless, onde cada requisição deve conter todas as informações necessárias para seu processamento; cacheável, permitindo que respostas sejam armazenadas para melhorar desempenho; interface uniforme, que padroniza a comunicação entre cliente e servidor; sistema em camadas, possibilitando a intermediação por proxies e gateways; e, opcionalmente, code on demand, no qual o servidor pode estender o comportamento do cliente através de código executável. Quanto mais dessas restrições uma aplicação respeita. Já o termo RESTful é utilizado para descrever sistemas ou APIs que aplicam, de forma consistente, os princípios do REST em seu design e implementação.

Para entender melhor para que serve REST, temos que estudar sua história. No final da década de 1990, Roy Fielding fazia parte do grupo responsável por padronizar o protocolo HTTP e projetar a arquitetura da Web. Ele foi um dos principais autores da versão 1.1 do HTTP. Essa época representava o início da Web moderna, e a comunicação entre sistemas na internet ainda era inconsistente — cada aplicação tinha seu próprio estilo de API. Fielding percebeu que, para a Web continuar evoluindo, seria necessário definir princípios arquiteturais claros e universais.

Em seus estudos, Roy Fielding analisou por que a Web (baseada em HTTP, URLs e HTML) conseguia escalar para milhões de usuários, enquanto outras arquiteturas falhavam. A partir dessas observações, ele definiu um conjunto de princípios arquiteturais que explicavam o sucesso da Web — e chamou esse estilo de REST.

Uma API REST utiliza os recursos da Web para representar entidades do sistema, como usuários, produtos, pedidos, entre outros. Cada recurso é acessado por meio de uma URL (Uniform Resource Locator), e as operações sobre esses recursos são realizadas através dos métodos HTTP.

Resumidamente, uma API REST funciona por meio de requisições (requests) e respostas (responses). A request é uma chamada HTTP feita pelo cliente, e a response é a resposta do servidor. O formato de dados mais comum nessa comunicação é o JSON (JavaScript Object Notation).

O REST foi um dos principais motivos da popularização do JSON. Antes dele, o padrão dominante era o SOAP, que utilizava XML para troca de dados. Comparado ao JSON, o XML é mais pesado — tanto em tamanho quanto em processamento —, pois exige um parsing mais complexo, além de aumentar o tamanho do texto com suas múltiplas tags, tornando o conteúdo mais verboso e de leitura difícil para humanos.

O JSON resolveu todos esses problemas e, desde então, tornou-se o content-type padrão nas APIs REST. No entanto, é importante destacar que o JSON não surgiu junto com o REST. Ele foi criado por Douglas Crockford e surgiu integrado ao JavaScript, como uma alternativa mais simples e eficiente ao XML. Com o tempo, acabou sendo adotado naturalmente pelas APIs REST devido à sua leveza e praticidade.

O estilo arquitetural SOAP (Simple Object Access Protocol) surgiu inicialmente como uma boa ideia em 1998, mas com o tempo acabou se tornando uma gambiarra completamente inviável. O SOAP foi uma evolução do protocolo XML-RPC (Remote Procedure Call usando XML), criado por Dave Winer no mesmo ano. O objetivo do XML-RPC era permitir que um servidor executasse funções em outro servidor, enviando parâmetros em XML e recebendo respostas também em XML — tudo isso utilizando o HTTP como transporte.

Ainda em 1998, a Microsoft e a IBM começaram a colaborar na evolução do XML-RPC, dando origem ao SOAP. O objetivo era criar um padrão mais formal, extensível e corporativo para a comunicação entre sistemas. Assim, o SOAP incorporou recursos como tipagem forte (baseada em XSD), suporte a headers (para autenticação, transações, etc.), extensibilidade (permitindo adicionar metadados sem quebrar o formato) e adesão aos padrões do W3C.

Tudo isso parecia uma boa ideia no início, mas desmoronou rápido. O SOAP acabou se tornando muito mais complicado do que precisava ser. Para fazer uma requisição simples era necessário lidar com estruturas XML enormes, envelopes, cabeçalhos e esquemas (XSD). Além disso, o SOAP exigia o uso de ferramentas específicas para gerar código, interpretar WSDLs (descrições de serviço) e lidar com as regras do protocolo — o que tornava tudo lento, burocrático e difícil de manter.

Por exemplo, veja como seria uma requisição simples apenas para buscar as informações de um usuário usando o SOAP: 


POST /UserService HTTP/1.1
Host: example.com
Content-Type: text/xml; charset=utf-8
SOAPAction: "getUser"

<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Header/>
  <soap:Body>
    <getUserRequest xmlns="http://example.com/user">
      <userId>123</userId>
    </getUserRequest>
  </soap:Body>
</soap:Envelope>

Agora compare com uma requisição REST que faz exatamente a mesma coisa:


GET /users/123 HTTP/1.1
Host: example.com
Accept: application/json

E a resposta do servidor REST seria bem mais simples e legível:


{
  "id": 123,
  "nome": "Adriel",
  "email": "adriel@example.com"
}

Essa diferença de simplicidade é o principal motivo da queda do SOAP e da popularização do REST. Enquanto o SOAP exigia XML pesado, contratos WSDL e múltiplas camadas de abstração, o REST usa os recursos nativos da Web (HTTP, URLs e JSON), tornando o desenvolvimento muito mais leve, rápido e acessível.

O SOUP não foi feito para a web moderna, essa arquitetura não respeita os métodos HTTP (GET, POST, PUT, DELETE), utiliza sempre POST nas requisições — tratando o http apenas como um túnel de transporte, é totalmente incompativel com caches HTTP e navegadores. Já REST, não tenta reinventar a roda, ela engloba todos esses recursos do protocolo HTTP — Talvez isso se deve ao fato do criador do Rest ser um dos principais fundadores da versão 1.1 do protoclo http.

Stateless vS Stateful

Uma das principais características do REST é o conceito de stateless. Isso significa que o servidor não armazena nenhuma informação sobre o estado da sessão do cliente entre as requisições.

Em uma arquitetura stateful, o servidor precisa manter informações de sessão do usuário — como dados de autenticação ou contexto da interação. Geralmente essas sessões são armazenadas em memória ou em bancos de dados rápidos, como o Redis. Assim, a cada requisição, o servidor consulta o estado previamente salvo para identificar o usuário e continuar o fluxo.

Já em uma API stateless, cada requisição deve conter todas as informações necessárias para ser processada: autenticação, parâmetros, contexto, entre outros. Um exemplo comum é o uso de tokens JWT, onde o cliente envia o token a cada chamada, permitindo que o servidor valide a requisição sem precisar manter qualquer estado entre uma interação e outra.

Atualmente, é muito mais recomendável implementar uma aplicação stateless do que stateful. Essa preferência se deve a vários fatores, tais como:

  • 1. Escalabilidade muito mais fácil: Como o servidor não mantém informações de sessão, qualquer requisição pode ser atendida por qualquer instância do backend. Isso facilita o balanceamento de carga, auto-escalonamento e a distribuição da aplicação em múltiplos servidores (microserviços).

  • 2. Redução de complexidade: Em sistemas stateless não é necessário sincronizar sessões, manter caches de estado ou usar bancos como Redis apenas para armazenar informações temporárias dos usuários. O servidor apenas processa requisições, tornando o sistema mais simples e robusto.

  • 3. Maior tolerância a falhas: Se uma instância cair, nenhuma sessão é perdida e outra instância pode assumir imediatamente as requisições. Isso aumenta a disponibilidade e a resiliência do sistema.

  • 4. Melhor desempenho em alto tráfego: Cada requisição é processada de forma independente, sem necessidade de consultar bancos de sessão ou memória compartilhada. Isso reduz a latência e melhora a performance.

Existem vários motivos pelos quais uma API stateless é muito mais vantajosa do que uma API stateful. No entanto, ainda há muitos backends web legados, especialmente aqueles baseados em MVC, que funcionam de forma totalmente stateful. Nesse tipo de aplicação, o servidor costuma controlar praticamente tudo: renderização de HTML, gerenciamento do fluxo da aplicação, autenticação baseada em sessão e outras responsabilidades centrais.

Uma observação importante sobre os termos: quando dizemos que um servidor é stateful, isso automaticamente significa que a API também é stateful. Embora servidor e API sejam conceitos diferentes, eles estão diretamente conectados — se o servidor mantém estado entre requisições, a API inevitavelmente deixará de ser stateless.

O próprio protocolo HTTP é stateless por natureza. Mas óbvio, nem tudo é tão simples... Existem algunas cenários especificos onde implementar uma aplicação web stateless não é o ideal.

Aplicações que necessitam de atualização em tempo real normalmente são stateful com o uso de WebSockets.

Outros exemplos clássicos de cenários stateful na Web incluem logins baseados em sessão, fluxos de múltiplas etapas (wizards) e sistemas legados que dependem de sessões no servidor.

Resumidamente, em grande parte das situações — provavelmente mais de 90% — um desenvolvedor web lidará com aplicações stateless em vez de stateful. Ainda assim, tudo depende do contexto. Compreender a diferença entre esses dois modelos é essencial para entender o funcionamento da Web moderna e REST.

GET

Agora vou demonstrar os recursos do REST na prática em uma api NODE com typescript, utilizando o framework Fastify e as demais bibliotecas já mencionadas.

Vou me concentrar mais na parte da api, ignorando coisas como banco de dados, ORMS e etc.

Uma API REST é orientada a recursos. Isso significa que, para acessar um determinado recurso (como registros de usuários, por exemplo), é necessário fazer uma chamada HTTP para uma URI específica.

Com o fastify vou criar as endpoints da API. Um endpoint é um ponto de entrada da API, definido pela combinação entre um método HTTP e uma URI específica. A URI (Uniform Resource Identifier) é o identificador que aponta para um recurso — como /users, /products/10 ou /auth/login. Cada endpoint da API possui sua própria URI, que indica ao cliente exatamente qual recurso está sendo acessado ou qual ação será executada. Em conjunto, método + URI determinam o comportamento do endpoint e como a aplicação irá responder a cada requisição.

Exemplo de uma API com alguns endpoints (cada linha exibe a URI da endpoint):


https://app.io/usuarios      → URL do recurso "usuarios"
https://app.io/usuarios/12   → URL do recurso "usuarios" com ID 12
https://app.io/boardgames    → URL do recurso "boardgames"

Para obter esses recursos em formato JSON, basta realizar uma requisição HTTP com o verbo GET.

GET - Route Params

Note a diferença entre a primeira e a segunda URL: a primeira retorna todos os usuários do banco de dados (ou de qualquer outra fonte que a API consulte), enquanto a segunda retorna apenas um usuário específico. Essa é a forma mais comum de filtrar um recurso específico na API, utilizando um Route Param.

Route params, como o próprio nome sugere, são parâmetros obrigatórios da rota e aparecem na própria URL. Em uma URL, tudo é sempre tratado como texto; ou seja, não existe tipagem na URL em si. A conversão do valor para um tipo específico (como número, boolean, etc.) é responsabilidade do framework da API.

Veja este exemplo:

Declarando uma endpoint com Node.js e Fastify

app é a instância da API utilizando Fastify.


app.get(
    '/usuarios/:id',
    {
        schema: {
            params: {
                type: 'object',
                properties: {
                    id: { type: 'number', minimum: 0 }
                },
                required: ['id']
            }
        }
    },
    async (request, reply) => userController.getUsuarioById(request, reply)
);

No código acima, estou declarando uma rota e definindo o tipo do parâmetro :id para que o Fastify faça a conversão e validação. Também foi definida uma regra (minimum: 0), e o próprio Fastify realizará a validação automaticamente, retornando uma exceção 400 caso a regra seja violada.

Em termos conceituais, em uma URL absolutamente tudo é representado como texto. Isso acontece porque a URL é apenas uma string codificada que o navegador envia ao servidor. Mesmo que você escreva valores que parecem numéricos, como /usuarios/10 ou /produtos/29.90, eles ainda chegam para a API como texto puro. Por exemplo, a URL /calcular/2/3 envia os valores "2" e "3" (como strings), e somente depois o framework converte esses dados para number, float, boolean ou qualquer outro tipo que você tenha definido.

Outro exemplo: se você enviar um nome com espaço, como /usuarios/joao silva, o navegador transforma isso em /usuarios/joao%20silva, que também é apenas texto codificado. Portanto, a URL por si só não entende tipos — ela transporta apenas strings, e cabe à API interpretar, validar e converter para o tipo apropriado.

Sobre boas práticas, recomenda-se que rotas sejam sempre nomeadas no plural quando representam coleções (ex.: /usuarios, /produtos), e que route params sejam escritos no singular, representando um item específico da coleção (ex.: /usuarios/:id).

Um "erro" muito comum em iniciante, é criar uma rota no singular quando a mesma possui um route param. Exemplo: 


https://app.io/usuarios     → URL do recurso "usuarios"
https://app.io/usuario/12   → URL do recurso "usuarios" com ID 12 (singular, prática incomum)

Isso não é um erro critico, ou algo que vai prejudicar sua API REST. Essas regras de boas práticas não foram definidas por alguém em especifico, não existe na tese de Roy Fielding um capitulo sobre "regras de nomeclaturas e mapeamentos" ou algo assim, isso tudo é um conjunto de convenções amplamente aceitas pela comunidade REST ao longo dos anos. Quando falamos de REST, quanto mais seguimos dentro desses padrões, melhor será a qualidade geral do código da API.

GET - Query Params

Outra forma de enviar parâmetros em uma requisição GET é usando query params. A principal diferença entre query params e route params está na obrigatoriedade. Route params fazem parte da estrutura da URL, portanto são obrigatórios. Já os query params são opcionais por natureza — eles só se tornam obrigatórios se você aplicar uma validação manual no seu backend.

Query params permitem enviar múltiplos parâmetros com diferentes finalidades, como filtros, paginação, ordenação, busca textual, entre outros. Por exemplo: 


https://app.io/usuarios?nome=adriel&orderby=idade
→ URL do recurso "usuarios", filtrando pelo nome "adriel" e ordenando por "idade".

Diferentemente dos route params, que identificam um recurso específico, os query params não alteram a identidade do recurso, apenas modificam a forma como o recurso será retornado — por exemplo, filtrando, ordenando ou paginando os resultados. Essa diferença é importante para a consistência do design de APIs REST.

Já me deparei com algumas aplicações trafegando tokens, senhas e outras informações sensiveis por query params. Obviamente isso é uma prática ruim — para dizer o minimo. URLs podem vazar em: históricos, logs de servidor, analytics, cache do navegador e etc. Query params serve para compartilhar informações opcionais cujo o objetivo é refinar, filtrar ou alterar a resposta.

Em query params é possivel compartilhar listas, os principais padrões são:


a) Repetindo a chave:

?categoria=acao&categoria=rpg&categoria=terror


b) Listas separadas por vírgula:

?categoria=acao,rpg,terror


c) Arrays com sintaxe:

?categoria[]=acao&categoria[]=rpg

Mas cuidado, evite transformar URLS em um payload com enorme quantidade de informação. Browsers normalmente suportam aproximadamente 2000 caracteres.

Outra prática comum em query params são os operadores explícitos como gte (maior ou igual), lte (menor ou igual), ne (diferente) ou in. Um formato bastante usado é o estilo inspirado em MongoDB: ?preco[gte]=100&preco[lte]=300. APIs mais simples usam convenções como ?minPreco=100&maxPreco=300. Exemplos: 


/usuarios?idade=30                    → igualdade
/usuarios?idade[gte]=18               → maior ou igual a 18
/usuarios?idade[lte]=65               → menor ou igual a 65
/produtos?preco[gte]=10&preco[lte]=50 → intervalo

É importante ressaltar que query params e route params não são funcionalidades criadas pelo estilo arquitetural REST. Na verdade, eles fazem parte da própria especificação de URLs definida pelas RFCs que padronizam a Web. Os route params são apenas trechos variáveis do caminho (path) da URL, definidos pela aplicação, enquanto os query params são parâmetros opcionais definidos pela RFC 3986 para transmitir dados na parte após o caractere ?. Portanto, REST apenas utiliza esses recursos já existentes na arquitetura da Web, mas não os define.

Declarando um endpoint com query params 

app.get(
    '/usuarios',
    {
        schema: {
            querystring: {
                type: 'object',
                properties: {
                    page: { type: 'integer', minimum: 1, default: 1 },
                    limit: {
                        type: 'integer',
                        minimum: 1,
                        maximum: 10,
                        default: 10,
                    },
                },
            },
        },
    },
    async (request, reply) => userController.getUsuarios(request, reply)
);

GET - Paginação

A paginação é uma técnica usada para controlar a quantidade de dados retornados em uma requisição. Sem ela, uma endpoint poderia enviar milhares de registros de uma só vez, o que causaria diversos problemas: lentidão no servidor, uso excessivo de banda, respostas muito grandes e um custo de processamento do payload muito alto no client.

Existem várias estratégias de paginação, mas as mais comuns são: Offset Pagination e Keyset Pagination.

Na offset pagination, o servidor pula um conjunto de registros (offset) e retorna um número fixo de itens (limit). offset e limit, são definidos como query params. O parâmetro page determina qual página de resultados será retornada, enquanto limit define a quantidade de itens por página.

Exemplo de URL com paginação:

GET https://api.exemplo.com/usuarios?page=2&limit=10

Nesse caso, a API retornaria a segunda página contendo até 10 usuários por página.

Vale ressaltar que a paginação não é um recurso obrigatório para a endpoint funcionar. Ou seja, caso o client não envie os parâmetros de paginação para a API, ela deve assumir valores padrão, como por exemplo: offset: 0 e limit: 50.

A paginação não entra em conflito com os demais parâmetros da query; portanto, a mesma endpoint pode conter simultaneamente paginação e filtros.

A fórmula para obter exatamente a página especificada pelo usuário é a seguinte: (page - 1) * limit. Após calcular esse valor, basta limitar o resultado pela quantidade definida em limit.

Veja um exemplo prático dessa fórmula aplicada no ORM Prisma:

async getUsuarios(page: number, limit: number) {
    return await prisma.usuario.findMany({
        skip: (page - 1) * limit, // pulando (page - 1) páginas
        take: limit,              // obtendo apenas a quantidade definida em limit
    });
}

A principal desvantagem dessa estratégia é a queda de desempenho em tabelas com grande volume de dados. Um valor muito alto de offset pode tornar o processamento lento, pois o banco precisa percorrer muitos registros antes de retornar os resultados. Essa abordagem é recomendada para datasets pequenos ou médios, oferecendo boa precisão de busca para o client.

Além da paginação, implementei o lookahead que é uma propriedade que sinaliza para o client a existência da próxima página (hasNextPage). 


async getUsuarios(
    page: number,
    limit: number
): Promise<UsuariosDto | null> {
    try {
        // Offset pagination with lookahead
        const usuarios = await this.usuarioRepository.getUsuarios(
            page,
            limit
        );
        const hasNextPage = usuarios.length == limit + 1;

        usuarios.pop();

        const usuariosDto = {
            data: usuarios,
            meta: {
                page,
                limit,
                hasNextPage: hasNextPage,
            },
        } as UsuariosDto;

        return usuariosDto;
    } catch (error) {
        console.error('Erro interno ao consultar os usuários.', error);

        throw new InternalError();
    }
}

Aqui, é onde aplicamos a "fórmula" da paginação, skipando os elementos e retornando apenas os especificados pelos parâmetros da paginação. 


async getUsuarios(page: number, limit: number) {
    return await prisma.usuario.findMany({
        skip: (page - 1) * limit,
        take: limit + 1,
    });
}

A outra estratégia de paginação popular em API REST é o Keyset Pagination (também chamada de Cursor based pagination). Essa estratégia consistem em navegar entre dos dados utilizando uma chave de referência — normalmente o valor de uma coluna ordenada e indexada, como id, created_at ou outra coluna monotonicamente crescente.

Veja esse exemplo de keyset pagination:


Endpoint:
GET /items?limit=10

Response:
{
  "items": [...],
  "nextCursor": "2025-01-10T14:03:22Z"
}

Próximo request:
GET /items?limit=10&cursor=2025-01-10T14:03:22Z

Query no backend:
SELECT *
FROM items
WHERE created_at > :cursor
ORDER BY created_at ASC
LIMIT 10;

Exemplo com ORM:
async getBoardgames(createdAt: Date | undefined, limit: number) {
    return await prisma.boardgame.findMany({
        where: {
            createdAt: {
                gte: createdAt,
            },
        },
        take: limit + 1,
    });
}

Perceba que a consulta é feita considerando apenas os dados criados após a última chave recebida (created_at > :cursor). Com isso, evita-se o problema de performance do offset pagination: nenhuma informação desnecessária é percorrida ou ignorada manualmente (skip), tornando a paginação muito mais eficiente.

Normalmente, a estratégia mais utilizada é a keyset pagination, por ser melhor em termos de perfomance.

GET - Caching

Cache é um recurso utilizado para melhorar o desempenho da aplicação, reduzindo o número de acessos ao banco de dados e acelerando o tempo de resposta das requisições.

Esse tema é extenso, para falar de maneira completa precisaria de um artigo somente dedicado a este tema, mas aqui vou apenas resumir um pouco dos conceitos e práticas.

Essa funcionalidade consiste em armazenar os dados requisitados em um banco de dados em memória. O mais comum é usar o Redis, devido à sua alta performance e popularidade. Assim, quando os mesmos dados são requisitados novamente, a aplicação consulta o cache em memória em vez do banco de dados.

O armazenamento em memória é essencial porque o acesso a dados na memória é muito mais rápido do que em disco. Isso torna o uso de cache uma solução eficiente para acelerar chamadas frequentes de dados.

Veja um exemplo do fluxo normal de uma chamada GET sem cache:


Cliente envia requisição GET → Servidor recebe a requisição → Servidor consulta o banco de dados → Banco retorna os dados → Servidor envia a resposta ao cliente

Agora, veja como o fluxo muda quando o cache está ativo:


Cliente envia requisição GET → Servidor verifica se os dados estão no cache (Redis) 
  → Se os dados estiverem no cache → Servidor retorna os dados diretamente ao cliente
  → Se não estiverem no cache → Servidor consulta o banco de dados → Banco retorna os dados → Servidor armazena os dados no cache → Servidor envia a resposta ao cliente

Perceba que o servidor consulta primeiro no cache, e somente se necessário consulta o banco de dados, esse é a funcionalidade base da maioria das estratégias de cache eficientes.

Existem várias formas de implementar cache em uma aplicação web, e a mais comum é a estratégia de cache-aside, cujo fluxo foi descrito acima.

No entanto, o próprio protocolo HTTP já fornece mecanismos de caching. Assim, além de gerenciar o cache nas camadas internas do servidor utilizando Redis, também podemos utilizar o Cache-Control para controlar o cache na camada da API. As APIs REST modernas geralmente combinam essas estratégias: Redis + Cache-Control (HTTP).

É importante entender a diferença entre essas duas abordagens, pois, apesar de frequentemente usadas juntas, elas são conceitos distintos. O cache-aside é uma estratégia do lado do servidor, que utiliza um banco de dados em memória como Redis ou Memcached para acelerar respostas. Essa solução é estritamente do servidor e não envolve o cliente.

Já o cache HTTP envolve o cliente de forma ativa. Nesse caso, os navegadores, proxies ou CDNs podem armazenar as respostas das requisições e reutilizá-las em chamadas futuras. O cabeçalho Cache-Control permite configurar regras como:

  • max-age: tempo em segundos que a resposta pode ser considerada válida.

  • public/private: define se a resposta pode ser armazenada por caches compartilhados (CDNs) ou apenas pelo navegador do cliente.

  • no-cache/no-store: instruções para que a resposta não seja reutilizada ou armazenada.

Combinando essas duas estratégias — cache-aside para o servidor e cache HTTP para o cliente — é possível construir APIs REST modernas que respondem rapidamente e reduzem significativamente a carga no banco de dados.

Mas, na prática, cerca de 90% das vezes os desenvolvedores web lidam apenas com o cache-aside, no lado do servidor. Já os mecanismos de cache HTTP são mais indicados para serem gerenciados pelo cliente ou por camadas intermediárias, como CDNs e proxies, através de cabeçalhos como Cache-Control e ETag. Tentar implementar a leitura de cache HTTP manualmente na sua API não é recomendado, pois esses mecanismos já são interpretados automaticamente por navegadores e CDNs, tornando desnecessária a lógica adicional no servidor.

Outro conceito importante de gerenciamento de cache é a invalidação de cache, isto é, dados que outrora eram úteis já estão defazados e precisam ser removidos/invalidados para que novos entrem no lugar.

Para entender isso visualize o seguinte cenário:


1. O cliente faz um GET /produtos/42
   - A requisição passa pelo cache.
   - Como existe um registro armazenado, o cache retorna a versão atual (mas antiga) do produto.
   - O cliente recebe os dados em milissegundos.

2. Um administrador atualiza o produto 42 no painel interno (ex.: altera preço de R$ 99 → R$ 89).
   - A API grava o novo valor no banco de dados.
   - Porém, o mecanismo de cache ainda possui a versão antiga.
   - Nenhuma invalidação foi disparada automaticamente.

3. O mesmo cliente faz outro GET /produtos/42 alguns segundos depois.
   - O cache continua retornando a versão antiga (preço R$ 99).
   - Do ponto de vista do cliente, o produto “parece” não ter sido atualizado.

4. Minutos depois, o TTL (Time To Live) do cache expira.
   - O cache marca o item como *stale* (obsoleto).
   - Na próxima requisição, o cache verifica que está expirado.
   - Ele encaminha a solicitação ao servidor para obter a versão mais recente.

5. O cliente faz um novo GET /produtos/42.
   - O cache detecta os dados como expirados.
   - A API retorna o preço correto (R$ 89).
   - O cache finalmente atualiza seu valor com a versão mais recente.

Esse fluxo mostra claramente:
- como dados úteis podem se tornar rapidamente obsoletos;
- como o cliente continua recebendo informações antigas até que ocorra invalidação ou expiração;
- a importância de mecanismos ativos de *cache invalidation* (ETags, versionamento, mensagens de evento etc.).

Em soluções modernas, ferramentas como Redis e Memcached já oferecem mecanismos nativos de TTL (Time to Live), permitindo que itens armazenados expirem automaticamente após um período definido.

Além do TTL, existem outras estratégias para lidar com dados inválidos. Algumas abordagens optam por atualizar o cache em vez de removê-lo completamente, enquanto outras combinam atualização e remoção conforme o cenário. Ainda assim, o uso de TTL continua sendo a técnica mais simples e amplamente adotada para garantir a expiração automática de informações.

Esses são apenas os fundamentos do tema. O universo de caching é bastante amplo, repleto de conceitos, padrões e estratégias diferentes. Explorar tudo em profundidade deixaria este artigo excessivamente longo e desviaria do foco principal, por isso abordei apenas o básico que todo programador backend deve saber sobre caching.

GET - HATEOAS / Hypermedia

Uma das ideias mais interessantes do REST original é que um cliente nunca deveria “adivinhar” quais ações são possíveis sobre um recurso; em vez disso, a própria resposta do servidor deveria incluir links navegáveis que descrevem o que pode ser feito a seguir. Esse conceito é chamado de HATEOAS (Hypermedia As The Engine Of Application State).

Na prática, isso significa que a resposta de um GET pode funcionar como uma pequena interface navegável, muito parecida com um documento HTML, mas usando JSON. Cada recurso retorna não só os dados, mas também metadados sobre como continuar a interação.

Exemplo simples:


{
  "id": 42,
  "title": "Pedido nº 42",
  "status": "Awaiting Payment",
  "links": [
    { "rel": "self", "href": "/orders/42" },
    { "rel": "customer", "href": "/orders/42/customer" },
    { "rel": "pay", "href": "/orders/42/pay" }
  ]
}

O cliente não precisa conhecer previamente a rota de pagamento ― a própria API entrega esse caminho de forma explícita e padronizada. Na teoria, isso permitiria que um cliente genérico navegasse por qualquer API REST, assim como um navegador faz com páginas web.

Embora esse conceito possa ser aplicado a qualquer método HTTP, ele costuma aparecer com mais frequência em GET. Na prática, porém, muitos desenvolvedores acabam não implementando esse recurso — e, em diversos casos, nem chegam a conhecer sua existência. Um HATEOAS bem estruturado reduz a necessidade de documentação extensa e cria uma comunicação mais direta entre cliente e servidor. Isso elimina comunicações humanas do tipo: “Ei, qual era mesmo o endpoint que retorna os dados do cliente relacionado ao pagamento?”, pois a própria resposta da API já aponta os próximos caminhos possíveis.

GET - Streams / Downloads

Outro ponto interessante — e pouco explorado — é que o GET não serve apenas para retornar JSON “fechado”. Ele também é o método mais adequado para conteúdos grandes, fluxos contínuos e transferências parciais, porque é seguro, idempotente e facilmente otimizado por navegadores, proxies e CDNs.

Trafegar arquivos é uma prática comum em aplicações web modernas: PDFs, imagens, vídeos, backups, relatórios e binários em geral. Em todos esses casos, o protocolo HTTP é utilizado para enviar os dados como um stream, ou seja, a conexão HTTP é estabelecida e os dados do arquivo são transmitidos de forma progressiva, em partes, enquanto o cliente os consome. Muitos programadores web se enganam ao pensar que ao declarar um GET com: content-type: application/pdf, o PDF será enviado de uma única vez, isso não existe.

Todo download de arquivo utilizando o protocolo HTTP se baseia na ideia de transferência progressiva de dados, onde o conteúdo é entregue como um fluxo contínuo de bytes ao cliente. Em muitos cenários — especialmente para arquivos grandes — essa transferência pode ser otimizada por meio da estratégia Range-based transfer, que permite solicitar apenas partes específicas de um recurso.

Um exemplo muito conhecido — e fácil de observar na prática — são os vídeos do YouTube. Embora um vídeo seja, conceitualmente, um arquivo, seus dados não são transferidos de uma única vez. Em vez disso, o conteúdo é trafegado como um stream de dados, sendo consumido progressivamente pelo cliente. Esse comportamento pode ser observado diretamente na ferramenta de desenvolvedor do navegador, na aba Network.

Durante a reprodução, é possível perceber sucessivas requisições ao endpoint /videoplayback. Essas requisições ocorrem ao longo do tempo e carregam os dados de vídeo e áudio necessários para manter o buffer do player preenchido:

Clique para expandir

Existem uma prática horrível, muito adotada — Principalmente com arquivos de imagens — que consiste em converter arquivos binários para Base64 e incluí-los dentro de um JSON. Além de poluir o payload e dificultar o cache, essa conversão aumenta o tamanho real do conteúdo em aproximadamente 33%. Sempre prefira HTTP streaming para a transferência de arquivos binários.

Quando o servidor conhece previamente o tamanho total do conteúdo a ser enviado, ele pode informar esse valor por meio do header Content-Length. Nesse cenário, o cliente sabe exatamente quantos bytes esperar e pode gerenciar melhor o progresso do download, exibir barras de progresso e otimizar o uso de memória. Por outro lado, quando o tamanho do conteúdo não é conhecido no momento em que a resposta é iniciada — como em streams, geração dinâmica de arquivos ou leitura progressiva de dados — o HTTP utiliza a transferência em partes (Transfer-Encoding: chunked, no header), na qual os dados são enviados em blocos sucessivos enquanto a conexão permanece aberta, sem a necessidade de informar o tamanho total antecipadamente.

Mas, no dia a dia, você não precisa se preocupar com essas propriedades no header ao transferir um arquivo, os próprios frameworks web já fazem isso por você, ou seja, ao retornar algo como: 


async getBoardgameRulebook(): Promise<ReadStream | undefined> {
    return fs.createReadStream(
        path.join(__dirname, '..', 'assets', 'rulebook-example.pdf')
    );
}

O própio FASTIFY vai se encarregar de incluir: Transfer-Encoding: chunked e Content-Length no headers da requisição.

E claro… sempre informe o Content-Type corretamente. Esse header indica ao cliente como os bytes recebidos devem ser interpretados. Por exemplo, application/json, text/html para páginas web, image/png ou image/jpeg para imagens, application/pdf para documentos PDF e video/mp4 para conteúdos de vídeo. Quando o tipo exato do arquivo não é conhecido ou não deve ser interpretado automaticamente pelo cliente, utilize o tipo genérico application/octet-stream, que representa um fluxo arbitrário de bytes.

POST

O objetivo do método POST é enviar dados ao servidor. Diferentemente do GET, em que o cliente solicita uma representação de um recurso, no POST o cliente já possui a informação e a transmite ao servidor para que ela seja processada.

Aqui, iniciantes se confundam com o fato de o POST também retornar uma resposta. Assim como o GET — e qualquer outro método HTTP — o POST sempre possui uma response, normalmente em formato JSON. Essa característica, porém, leva a um erro conceitual frequente: utilizar POST para obter dados do servidor. Embora isso funcione tecnicamente, está longe de ser a abordagem correta. O POST não deve ser usado como mecanismo de consulta; seu papel principal é a criação ou submissão de dados no servidor, sendo normalmente associado a operações de INSERT.

Também vale ressaltar que muitos dos conceitos apresentados no tópico anterior sobre o GET — como query params, route params, paginação e hypermedia — podem ser utilizados normalmente em requisições POST, ainda que alguns desses recursos sejam mais comuns no contexto de GET. A principal diferença semântica entre POST e GET no HTTP é que, no primeiro, é permitido o envio de um payload no body da requisição, enquanto no segundo o corpo da requisição não é utilizado para esse fim.

Assim como fiz no tópico sobre práticas comuns do GET, vou descrever as situações e práticas comuns de POST que um programador web backend lida diariamente.

POST - Mapeamento e validação de dados

Quando o cliente envia dados no corpo da requisição, esses dados devem ser mapeados e validados. O protocolo HTTP não realiza nenhum tipo de validação e mapeamento, ele apenas transporta os dados. A responsabilidade de converter o corpo em objetos ou classes no servidor fica a cargo do framework web (no meu caso, o Fastify).

Sobre mapeamento, existem diversas bibliotecas que fazem o parsing do corpo da requisição em um DTO (Data Transfer Object). No caso do Fastify, a biblioteca padrão para isso é o AJV.

Uma boa biblioteca de parsing de API web não fornece apenas o mapeamento (transformar JSON em uma classe), mas também recursos de validação de dados. Esses recursos permitem validar regras como tipo de dados, valores mínimos e máximos para dados numéricos, campos obrigatórios, entre outros. Quando a biblioteca de parsing é totalmente integrada ao framework da API, essas validações já retornam erros HTTP da família 400 sempre que algum dado enviado pelo cliente não corresponde às regras estipuladas no mapeamento.

Para entender melhor veja, esse mapeamento de dados da enpoint /usuario para criação de um novo usuário no fastify: 

                                
app.post(
    '/usuarios',
    {
        schema: {
            body: {
                type: 'object',
                required: ['email'],
                properties: {
                    nome: { type: 'string' },
                    email: { type: 'string' },
                },
            },
        },
    },
    async (request, reply) => userController.createUsuario(request, reply)
);

Perceba, que eu consigo definir o tipo das propriedades, e definir a obrigatóriedade, atráves da propriedade required. Se o cliente, enviar um valor númerico no nome, o AJV avisa o fastify que uma regra não foi cumprida, e o próprio fastify se encarrega de retornar um erro 400 para o cliente. Tudo automático, não foi necessário validar manualmente dentro dos meus métodos controller ou service.

Todo framework web moderno, já fornece esse tipo de mapeamento e validação com retorno de exceção automático. Definitivamente não é uma boa prática validar corpo de requisição manualmente. Sendo assim o desenvolvedor deve apenas se preocupar com as validações que são necessárias, as validações que envolvem as regras de negócio do sistema.

Após definir o schema do body da requisição na endpoinnt POST /usuario, eu criei uma interface DTO que mapeia as propriedades desse schema, veja: 


export interface UsuarioDto {
    nome: string | null | undefined;
    email: string;
    created?: boolean | undefined;
}

Utilizo essa interface na minha controller e service, veja esse exemplo em usuarioController: 


async createUsuario(
    request: FastifyRequest,
    reply: FastifyReply
): Promise<undefined> {
    try {
        const newUsuario = request.body as UsuarioDto;

        const usuario = await this.usuarioService.createUsuario(newUsuario);

        return reply.code(usuario?.created ? 201 : 200).send(usuario);
    } catch (error) {
        this.throwResponseException(error, reply);
    }
}

request.body é recebido como um any, mas eu forço a tipagem para UsuarioDto. Utilizar DTOs é uma boa prática em métodos POST. Entenda DTO como uma classe "burra", que não possui nenhuma funcionalidade, apenas define as propriedades e seus tipos. O próprio nome já sugere que é uma classe apenas para transferencia de dados (Data Transfer Objects).

É importante comentar sobre DTOs, pois programadores iniciantes tendem a utilizar a mesma classe de Model do ORM para mapear os dados na controller. Esse é um erro muito comum. Para entender melhor, veja as duas classes abaixo:

Interface TypeScript que define o modelo da tabela USUÁRIO, utilizada pelo ORM: 


export interface UsuarioModel {
    id: number;
    nome: string;
    email: string;
    senha: string;
    criadoEm: Date;
    atualizadoEm: Date;
}

Interface TypeScript que define o DTO para o POST, utilizado na criação de um novo usuário: 


export interface UsuarioDto {
    nome: string;
    email: string;
    senha: string;
}

Perceba que as duas interfaces possuem algumas propriedades semelhantes, mas o DTO não inclui campos como id, criadoEm ou atualizadoEm. À primeira vista, pode parecer que não faz sentido criar um UsuarioDto e que seria mais simples usar diretamente o UsuarioModel na controller. No entanto, isso é um erro comum de programadores iniciantes.

Separar Model e DTO faz sentido por vários motivos: o DTO define exatamente o que a API espera receber ou enviar, protegendo campos sensíveis como senhas ou IDs de serem modificados diretamente pelo cliente. Além disso, facilita a validação e o mapeamento de dados, mantém a camada de persistência desacoplada da camada de apresentação e torna o código mais seguro, previsível e fácil de manter. Essa prática é especialmente importante em APIs REST ou GraphQL, onde o contrato de dados precisa ser claro e consistente.

Para simplificar: não misture camadas. Models fazem parte da camada de banco de dados e do servidor, e devem ser utilizados apenas nas classes responsáveis pela persistência, como os repositories. Nunca devem ser usados diretamente na camada de API, como nos métodos da controller. Mais adiante, abordarei com mais detalhes o conceito de camadas e reforçarei a importância de manter cada responsabilidade separada.

POST - Processamento assíncrono

APIs modernas utilizam async/await porque a maior parte do processamento envolve operações de I/O, como acesso a banco de dados, leitura de arquivos e chamadas para outros serviços. Essas operações são inerentemente lentas quando comparadas à execução de código em memória, e executá-las de forma síncrona bloquearia o servidor. O uso de processamento assíncrono permite que a aplicação continue atendendo outras requisições enquanto aguarda a conclusão dessas operações.

Pensamento clássico: "Se eu usar async/await, posso fazer qualquer coisa dentro dos métodos da endpoint.". Esse pensamento está completamente errado, async/await não paraleliza CPU ele apenas libera a thread enquanto espera I/O, ele não resolve problemas causados por tarefas computacionalmente pesadas. Operações intensivas de CPU, como cálculos complexos, compressão de dados ou loops extensos, continuam sendo executadas de forma síncrona dentro da thread de execução.

Para resolver esse problema e evitar o bloqueio do event loop no Node.js, é comum adotar a estratégia de filas de processamento. Nesse modelo, a API limita-se a receber e validar a requisição, delegando a execução de tarefas pesadas para processos em segundo plano. Dessa forma, o servidor HTTP permanece responsivo, enquanto o processamento é realizado de maneira assíncrona e desacoplada do ciclo de vida da requisição.

Esse tipo de problema ocorre principalmente quando estamos lidando com sistemas de envio de e-mails, mensagens e etc. Para entender, visualize esse fluxo que não possui fila de processamento: 


Cliente
   |
   v
API recebe a requisição
   |
   v
API processa toda a lógica (envio de e-mail)
   |
   v
API aguarda o serviço de e-mail responder
   |
   v
Resposta enviada ao cliente (200 OK ou erro)

Nesse cenário, o cliente precisa aguardar todo o processamento terminar. Caso o serviço de e-mail esteja lento ou indisponível, a requisição ficará bloqueada, aumentando o tempo de resposta e a chance de falhas.

Agora veja o mesmo fluxo utilizando uma fila de processamento (REST + Queue): 


Cliente
   |
   v
API recebe a requisição
   |
   v
API valida os dados
   |
   v
API adiciona a mensagem na fila
   |
   v
Resposta imediata ao cliente (202 Accepted)

-------------------------

Worker / Consumidor
   |
   v
Lê mensagem da fila
   |
   v
Processa a lógica (envio de e-mail)
   |
   v
Comunica o serviço externo
   |
   v
Finaliza o processamento

Com a fila, a API responde rapidamente ao cliente, delegando o trabalho pesado para um worker. Isso torna o sistema mais resiliente, escalável e desacoplado, evitando que falhas ou lentidão em serviços externos impactem diretamente o usuário final.

Nos últimos anos, tornou-se popular o uso de aplicações de mensageria, como RabbitMQ. É importante entender a diferença entre fila de processamento e mensageria: Embora ambos os conceitos permitam processamento assíncrono, filas de processamento e mensageria possuem objetivos diferentes. A fila de processamento é utilizada para delegar a execução de tarefas a um worker, geralmente dentro do mesmo sistema, garantindo que operações custosas sejam processadas em segundo plano sem bloquear a requisição HTTP. Já a mensageria tem como foco a comunicação entre serviços, onde eventos ou mensagens são publicados para um broker e consumidos por um ou mais sistemas de forma desacoplada, sendo muito comum em arquiteturas de microserviços.

Ou seja, se a sua tarefa é implementar um processamento assíncrono, sem depender de outros serviços, tais como: processamento pesado no banco de dados, geração de relatório, envio de emails, tarefas que podem falhar e precisar de retry, entre outros cenários, a escolha mais adequada é utilizar uma fila de processamento.

Já a mensageria é mais indicada quando há a necessidade de comunicação entre sistemas ou serviços distintos, onde um evento precisa ser propagado para que outros serviços reajam a ele. Esse modelo é comum em arquiteturas de microserviços, permitindo maior desacoplamento, escalabilidade e independência entre os consumidores da mensagem.

POST - Webhook

Outro recurso fortemente ligado ao tema de processamento assíncrono são os webhooks — não confundir com WebSockets, pois são conceitos completamente diferentes.

Um webhook é um mecanismo de notificação baseado em eventos, no qual um sistema envia automaticamente uma requisição HTTP para outro sistema quando algo acontece. O webhook pode ser entendido como o oposto do polling HTTP.

Entenda webhook como: “me avisa quando acontecer”.
Entenda polling como: “vou te perguntar a cada 5 segundos se algo aconteceu”.

Faz sentido utilizar webhooks quando é necessário disparar um evento assíncrono em outra aplicação, como por exemplo: pagamento aprovado ou reprovado, confirmação de e-mail de um usuário, finalização de um processamento em background, entre outros.

Na prática, um webhook é apenas uma chamada HTTP para outra API, sendo predominantemente utilizado para comunicação backend → backend, normalmente em cenários de integração entre sistemas, como: Gateway de pagamento → e-commerce. Webhooks não são pensados para comunicação direta com frontend.

Webhooks quase sempre são implementados utilizando endpoints do tipo POST. Embora seja tecnicamente possível utilizar outros métodos HTTP, na prática isso raramente faz sentido.

Por ser amplamente utilizado em integrações entre APIs, muitos desenvolvedores acabam atribuindo aos webhooks uma complexidade que não existe. Um webhook é apenas uma chamada HTTP para outra API, sinalizando que algo aconteceu — nada além disso.

Existem alguns artigos que defendem o uso de polling HTTP, mas particularmente, eu não gosto dessa prática. Além de consumir muita banda desnecessária, para realizar as chamadas http constantes, também consome processamento, pois cada request trafega dados de autenticação, permissão e etc, que precisa ser processado. E eu considero algo muito feio de implementar, todas as vezes que implementei polling http, foi por que o serviço que eu estava integrando me obrigou a usar polling, isto é, ele não liberava um webhook para minha aplicação se comunicar.

A minha opinião é: Para notificação de eventos entre API's, prefira webhook sempre que possível, evite polling http ao máximo.

POST - Upload / Multipart

Para upload de arquivos, é possível trafegar apenas os dados do arquivo, basta declarar uma endpoint com application/pdf (caso seja um arquivo PDF). ou então um octet-stream, que é um content-type "coringa", que não especifica o tipo do arquivo, é utilizado apenas para trefegar bytes.

Mas existe um Content-Type interessante, pouco utilizado e muitas vezes mal compreendido, que é o multipart/form-data. Com esse formato é possível enviar dados binários + metadados na mesma requisição, como no caso do cadastro de um produto ou boardgame, onde informações como nome, descrição e preço são enviadas junto com a imagem associada ao item.

Para simplificar, eu costumo pensar assim:


Preciso enviar apenas um arquivo
→ application/pdf, image/png, image/jpeg ou application/octet-stream

Preciso enviar apenas dados estruturados (texto, números, booleanos)
→ application/json

Preciso enviar arquivo + dados
→ multipart/form-data

Vamos entender um pouco sobre multipart/form-data: é um formato de codificação de requisições HTTP definido pelo padrão MIME, usado quando precisamos enviar múltiplos tipos de dados no corpo da mesma requisição.

Em vez de enviar tudo como um único bloco (como no application/json), o corpo da requisição é dividido em partes independentes (parts), cada uma contendo: seus próprios headers seu próprio conteúdo. Essas partes são separadas por um boundary.

JSON não foi feito para trafegar arquivos, soluções toscas como converter em BASE64 (Já discutido no tópico GET - STREAM) não funcionam bem. JSON é um formato textual que foi criado para representar dados estruturados legíveis, não para transportar dados binários brutos.

Por isso, multipart resolve o problema, pois o formato multipart suporta http streaming, ou seja, o arquivo é lido em chunks de rede, e não é necessário carregar todo o binário em memória. o servidor processa os dados enquanto recebe. Isso é essencial para uploads grandes e torna a API mais escalável.

Veja esse exemplo de corpo multipart de um arquivo de imagem (apenas uma representação visual, no body tudo isso é enviado em binário): 


------WebKitFormBoundaryXyZ123
Content-Disposition: form-data; name="title"

Capa do Boardgame
------WebKitFormBoundaryXyZ123
Content-Disposition: form-data; name="image"; filename="cover.png"
Content-Type: image/png

<bytes binários da imagem>
------WebKitFormBoundaryXyZ123--

No exemplo acima, o boundary (------WebKitFormBoundaryXyZ123) é um delimitador utilizado para separar cada parte do corpo da requisição multipart/form-data. Ele é definido no header Content-Type da requisição e indica onde uma parte começa e termina.

Cada parte do corpo contém seus próprios headers, como Content-Disposition e Content-Type. Esses headers descrevem os metadados daquela parte específica, informando, por exemplo, o nome do campo do formulário, o nome do arquivo e o tipo do conteúdo enviado.

Já o conteúdo corresponde aos dados propriamente ditos daquela parte. No caso de campos de texto, trata-se do valor textual enviado; no caso de arquivos, são os bytes binários do arquivo. Todo esse conteúdo é transmitido como um fluxo de bytes, sendo interpretado pelo servidor de acordo com as regras do formato multipart.

Em resumo, o multipart/form-data não é apenas um formato para upload de arquivos, mas um mecanismo que permite transportar diferentes tipos de dados de forma eficiente, segura e compatível com streaming, sendo a escolha ideal quando arquivos e metadados precisam coexistir na mesma requisição.

POST - Idempotência

Uma operação é idempotente quando executá-la uma ou várias vezes produz o mesmo efeito final no servidor.

Em resumo: Repetir a mesma requisição não altera o resultado após a primeira execução bem-sucedida. Exemplo genérico: 


Executar 1 vez → estado final X 
Executar 10 vezes → continua estado final X

Por padrão, métodos do tipo POST não são idempotentes. O método POST normalmente é usado para criação de recursos, e cada chamada tende a gerar um novo efeito colateral: 


POST /orders

Se chamado duas vezes:
1 vez → Pedido #101 criado

2 vex → Pedido #102 criado

➡️ Dois efeitos distintos → não idempotente

É possivel tornar o POST idempotente, desde que a aplicação implemente esse comportamento explicitamente na api. Entenda que tornar um POST idempotente não é só uma boa prática, mas sim uma solução de um problema de duplicidade de operação.

Nem todo POST necessita de idempotência, cada caso deve ser analisado. Mas no geral a idepotencia deve ser implementada quando a operação não pode ser duplicada, principalmente quando é uma operação critica, como por exemplo uma API de pagamentos, criação de pedidos e etc.

Idempotência resolve principalmente problemas causados por repetições involuntárias de requisições, como o um cliente que está duplicando request, usuário que clica diversas vezes no button "submit" do formulário, uma comunicação http que se perdeu e o http-client possui retry automático, e etc.

O padrão mais utilizado para implementar idempotência em API REST é a estratégia de chave de idempotência.

Essa estratégia consiste em o cliente gerar uma chave única (chamada de idempotency-key) e envia-la como metadado da requisição.

Normalmente essa chave é um UUID, é obrigatória, e deve ser enviada no HEADERS da requisição, pois é um metadado. Quando essa informação não é enviada, a API retorna 400 ou 422 para o client.

Segue o fluxo dessa estratégia é assim:


POST /orders
Headers:
  Idempotency-Key: 8f1a2c9e-...

Funcionamento típico:

1. Cliente gera um UUID e envia no header Idempotency-Key.

2. A API verifica se essa chave já foi utilizada.

3. Se a chave NÃO existir:
   - Processa a requisição
   - Persiste o resultado associado à chave
   - Retorna a resposta ao cliente

4. Se a chave JÁ existir:
   - Não executa a operação novamente
   - Retorna exatamente o mesmo resultado da primeira execução

Para que o servidor retorne o resultado da primeira execução, é necessário armazenar essa Idempotency-Key junto com o resultado.

Normalmente esse armazenamento é feito em um banco de dados de rápido acesso e que possua funcionalidade de TTL, pois idempotency-key é uma informação que deve ser descartada depois de um tempo, como um Redis ou Memcached.

No meu caso, optei por utilizar redis. Com redis o fluxo do exemplo, mais detalhado é assim: 


POST /orders
Headers:
  Idempotency-Key: 8f1a2c9e-...

Funcionamento típico:

1. A API recebe a requisição e extrai o header Idempotency-Key.

2. A API consulta o Redis usando a chave de idempotência:
   - GET idempotency:8f1a2c9e-...

3. Caso a chave NÃO exista:
   - Cria um registro no Redis com status "processing"
   - Define um TTL para essa chave
   - Executa a lógica principal (ex: criação do pedido)
   - Ao final, atualiza o registro com:
     - status: "completed"
     - response: payload da resposta
     - http_status: 201

4. Caso a chave JÁ exista:
   - A API NÃO executa a lógica novamente
   - Retorna exatamente a mesma resposta armazenada
     (mesmo status HTTP e mesmo payload)

5. Após o tempo definido no TTL:
   - A chave é automaticamente removida do Redis
   - A operação pode ser executada novamente com uma nova Idempotency-Key

É importante entender que idempotência não é um recurso do HTTP ou uma regra do REST. idempotência, é considerado um contrato entre a api e o cliente, ou seja, uma regra definida entre ambas as partes para garantir que uma operação não seja duplicada.

PUT / PATCH / DELETE

Decidi agrupar os métodos PUT, PATCH e DELETE em um único tópico, pois eles não possuem mecanismos específicos tão complexos quanto GET ou POST. Ou seja, nesses método não costuma ser comum mecanismos como idempotencia, caching, content-negociation e etc. Ainda assim, são métodos fundamentais e merecem destaque, com o objetivo de esclarecer suas funcionalidades, diferenças e boas práticas de uso, concluindo assim os cinco métodos HTTP mais utilizados em APIs REST.

Começando pelo PUT, esse método HTTP tem como objetivo alterar um recurso já existente. Seu uso ideal consiste em enviar toda a representação do recurso no corpo da requisição para que a atualização seja realizada.

Para ficar mais claro, imagine um endpoint PUT /usuarios/:id, responsável por modificar os dados de um usuário na base de dados. Nesse caso, todas as informações do usuário devem ser enviadas no payload, como email, username, foto, entre outras. Ou seja, o PUT representa uma atualização completa do recurso.

Já o PATCH possui um propósito diferente. Um endpoint do tipo PATCH indica que apenas algumas propriedades específicas do recurso serão atualizadas. Enquanto o PUT realiza uma atualização completa, o PATCH representa uma atualização parcial. Por exemplo, se existir um endpoint destinado exclusivamente à atualização da foto do usuário, o método PATCH é o mais indicado, pois não há necessidade de enviar todas as demais informações, como email ou username.

Na prática, é comum ver o PUT sendo utilizado de forma imprecisa sempre que o objetivo é alterar um recurso. Para evitar esse tipo de ambiguidade, costumo utilizar a seguinte pergunta como regra prática: é necessário enviar toda a representação do recurso para realizar essa atualização? Se a resposta for sim, o PUT é a melhor escolha; caso contrário, o PATCH tende a ser mais adequado.

Vale ressaltar que é uma boa prática retornar o dado atualizado depois que a alteração foi concluida com sucesso em um PUT ou PATCH. Assim o frontend (ou quem estiver consumindo), já receberá no response a informação em seu estado final.

Por fim, temos o método DELETE, cujo objetivo é remover um recurso do servidor. Assim como o PUT, o DELETE atua sobre um recurso específico, geralmente identificado por um id na URL, como em DELETE /usuarios/:id.

Quando um DELETE é executado com sucesso, espera-se que o recurso não esteja mais disponível para futuras consultas. Em APIs REST, é comum que esse método retorne um 204 No Content, indicando que a operação foi realizada corretamente, mas sem a necessidade de retornar um corpo na resposta.

Conceitualmente, o DELETE é considerado um método idempotente (assim como o PUT, PATCH e GET). Isso significa que múltiplas requisições DELETE para o mesmo recurso produzem o mesmo efeito final: o recurso deixa de existir. Caso o recurso já tenha sido removido, a API pode retornar um 404 Not Found ou simplesmente manter o 204, dependendo da estratégia adotada.

É importante destacar que, na prática, o DELETE nem sempre representa uma remoção física do dado. Em muitos cenários, especialmente quando há requisitos de auditoria ou histórico, utiliza-se o conceito de soft delete, onde o recurso é apenas marcado como inativo, permanecendo armazenado na base de dados, mas indisponível para uso normal da API.
